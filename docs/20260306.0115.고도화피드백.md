## 검토 결과

### 잘 된 점

기존 5섹션 19헤딩 구조를 유지하면서 Phase η 내용을 자연스럽게 녹여낸 점, 그리고 부록 B(서브태스크 상세)와 부록 C(설계 결정 변경 이유)를 추가하여 "왜 계획이 바뀌었는지"를 명시적으로 기록한 점이 좋습니다. 특히 부록 C의 란타나이드→4d TM 변경 근거(LANL2DZ ECP 커버리지 확인)는 JOSS 리뷰어가 물어볼 수 있는 질문에 대한 선제 답변이 됩니다.

### 수정이 필요한 문제점

**1. 수치가 "예상치"와 "확인치"가 혼재되어 있습니다.**

Phase η는 아직 WSL2에서 실행되지 않았습니다. 그런데 보고서에서 "130+ passed 예상", "60 벤치마크 케이스" 등이 마치 확인된 것처럼 서술된 부분이 있습니다. 구체적으로:

§2.5 테스트 통과 행에서 `108+ passed (Phase η 코드 미적용 시), 130+ passed (Phase η 전체 적용 시 예상)`은 괜찮은데, §4.1에서 "108+개 테스트(자동화), 60개 벤치마크 케이스"라고 단정적으로 쓴 것은 아직 실행 전이므로 "~60 케이스 (η-4 확장 벤치마크 포함 시)"처럼 구분해야 합니다. §4.5의 "60 케이스 중 에너지·IBO 수·전하 보존 모두 검증"도 마찬가지입니다. 확장 벤치마크는 아직 돌려보지 않았으므로 "검증 예정"이 정확합니다.

**2. 경쟁 비교표(§3.4)에 마크다운 문법 오류가 있습니다.**

테이블 마지막 세 행에 `| ← updated`가 행 바깥에 붙어 있어서 렌더링 시 깨집니다:

```
| 스칼라 상대론 (X2C)    |    ✅     |    ❌    |     ❌      |    ❌     |   ❌    | ✅ (스크립트) | ← updated |
```

`← updated`를 셀 안에 넣거나 행 끝 주석으로 처리해야 합니다. 이 표는 7열(도구 6개 + 기능명)인데 마지막 세 행만 8열이 되어 렌더러에 따라 깨집니다.

**3. 버전 날짜 불일치가 있습니다.**

보고서 상단에 `2026‑03‑06`으로 되어 있는데, CHANGELOG.md와 PROGRESS.md, 그리고 실제 git 태그는 `2026‑03‑05`입니다. Phase η가 아직 실행되지 않았으므로 날짜를 `2026‑03‑05 (Phase η 계획 반영)`으로 수정하거나, 실행 후 실제 날짜로 업데이트하는 것이 맞습니다.

**4. §2.4.2에서 란타나이드 서술과 실제 구현 범위가 모순됩니다.**

§2.4.2에 "란타나이드/악티나이드 원소(Z=57–71, 89–103)가 감지되면" 폴백이 적용된다고 썼는데, 부록 C에서는 란타나이드를 포기하고 4d TM으로 변경했다고 명시합니다. 감지·경고 로직 자체는 코드에 남아있으므로 틀린 것은 아니지만, 독자가 "란타나이드가 지원되는 것인가?"라고 오해할 수 있습니다. "란타나이드가 감지되면 경고를 발생시키며, 현재 버전에서는 4d TM(Zr, Mo)까지만 벤치마크 검증되었다"라고 범위를 명확히 해야 합니다.

**5. AI 사용 고지에 Gemini가 추가되었는데 근거가 불명확합니다.**

§4.6에서 "Anthropic Claude (Claude Opus 4 계열), Google Gemini"라고 되어 있습니다. 이 보고서를 Gemini 3.1 Pro가 작성했으므로 기술적으로 맞지만, Gemini의 구체적 사용 범위(이 보고서 통합 작성)를 명시하는 것이 JOSS AI 공개 정책에 부합합니다. 단순히 이름만 나열하면 "어디에 뭘 썼는지" 불분명합니다.

### 구조적 개선 제안

**부록 A가 없습니다.** "부록: Phase 이력 요약" 다음에 바로 "부록 B"가 나옵니다. "부록: Phase 이력 요약"을 "부록 A"로 명명하거나, 부록 B/C를 A/B로 변경하는 것이 일관적입니다.

**§2.3.1과 §2.4.1, §2.4.2는 Phase η 전용 서브섹션인데**, 향후 Phase θ 이후에도 계속 버전별 서브섹션을 추가하면 구조가 비대해집니다. Phase η 내용을 기존 §2.3(백엔드 상세)과 §2.4(파이프라인)에 통합 서술하고, 변경 이력은 부록으로 분리하는 것이 장기적으로 유지보수하기 쉽습니다. 다만 이것은 지금 당장의 문제는 아닙니다.
