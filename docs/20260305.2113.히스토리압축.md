# QCViz-MCP 프로젝트: 지금까지 만든 것, 현재 상태, 앞으로 할 일

---

## 이 프로젝트가 대체 뭔가

한 문장으로 말하면, "AI한테 '이 분자의 결합 오비탈 계산해서 3D로 보여줘'라고 말하면 진짜 해주는 서버"를 만들고 있습니다.

좀 더 풀어서 설명하겠습니다. 양자화학에서 분자의 전자 구조를 이해하려면 오비탈이라는 개념을 씁니다. 그중에서도 Intrinsic Bond Orbital, 줄여서 IBO라는 것이 있는데, 이건 Gerald Knizia라는 사람이 2013년에 제안한 방법으로 분자 내 결합을 직관적으로 시각화할 수 있게 해줍니다. 예를 들어 물 분자에서 산소의 고립 전자쌍이 어디에 있는지, O-H 결합 전자가 공간상 어떤 모양인지를 3D 등치면(isosurface)으로 보여주는 겁니다.

문제는 이 과정이 꽤 번거롭다는 것입니다. 연구자가 직접 PySCF라는 양자화학 프로그램으로 SCF 계산을 돌리고, IAO(Intrinsic Atomic Orbital)를 만들고, 그걸 IBO로 국소화하고, cube 파일이라는 3D 격자 데이터를 생성하고, 그걸 또 IboView나 VMD 같은 별도 프로그램에서 열어야 합니다. 단계마다 다른 도구, 다른 파일 포맷, 다른 명령어를 써야 하니까 연구 흐름이 끊깁니다.

우리가 만든 QCViz-MCP는 이 전체 과정을 하나의 함수 호출로 압축합니다. 그것도 그냥 함수 호출이 아니라, MCP(Model Context Protocol)라는 표준 프로토콜을 통해서요. MCP는 AI 에이전트가 외부 도구를 호출할 수 있게 해주는 프로토콜인데, 쉽게 말하면 ChatGPT나 Claude 같은 AI가 "이 서버에 있는 도구를 쓰겠다"고 선언하고 실제로 실행할 수 있게 해주는 표준 규격입니다. 그러니까 최종적으로는 Claude Desktop에서 "물 분자의 IBO를 계산해서 시각화해줘"라고 자연어로 말하면, Claude가 QCViz-MCP 서버의 `compute_ibo` 도구를 호출하고, 서버가 PySCF를 돌려서 IBO를 계산하고, cube 데이터를 생성하고, py3Dmol로 3D HTML을 만들어서, 사용자에게 결과를 돌려주는 겁니다.

이게 왜 의미가 있냐면, 기존에 MCP 서버 형태로 분자를 다루는 프로젝트들이 몇 개 있긴 합니다. PyMOL을 원격 조작하는 것, ChimeraX를 제어하는 것, ASE로 원자 시뮬레이션을 하는 것 등이요. 하지만 이것들은 전부 "분자 구조" 수준에 머물러 있습니다. 원자 위치를 보여주거나, 입력 파일을 생성해주거나 하는 정도입니다. "전자 구조" 수준, 그러니까 오비탈을 계산하고 시각화하는 MCP 서버는 우리가 전수조사한 바로는 세상에 아직 없습니다. QCViz-MCP가 그 첫 번째입니다.

## 기술적으로 어떻게 생겼는가

프로젝트의 구조를 설명하겠습니다. Python 패키지이고, `src/qcviz_mcp/` 아래에 모든 소스 코드가 들어갑니다.

핵심은 백엔드(backend)라는 계층입니다. 네 종류가 있습니다. 첫째, PySCF 백엔드는 실제 양자화학 계산을 담당합니다. Hartree-Fock이든 DFT든 SCF 계산을 돌리고, IAO를 생성하고, IBO로 국소화하고, cube 파일 데이터를 만듭니다. 둘째, cclib 백엔드는 다른 양자화학 프로그램의 출력 파일을 파싱합니다. ORCA, Gaussian, GAMESS 등 16개 프로그램의 출력에서 에너지, MO 계수, 진동 주파수 같은 데이터를 뽑아냅니다. 셋째, py3Dmol 백엔드는 시각화를 담당합니다. 분자 구조나 오비탈 등치면을 인터랙티브 3D HTML로 렌더링합니다. 넷째, ASE 백엔드는 분자 구조 파일의 포맷 변환을 담당합니다. XYZ를 CIF로, PDB를 POSCAR로 바꾸는 식입니다.

이 네 백엔드 위에 BackendRegistry라는 자동 탐지 시스템이 있습니다. 서버가 시작될 때 "PySCF가 설치되어 있나? 있으면 등록, 없으면 건너뛰기"를 자동으로 합니다. 덕분에 PySCF가 없는 Windows 환경에서도 cclib 파싱이나 ASE 변환 같은 기능은 정상적으로 동작합니다.

백엔드 위에 MCP 도구(tool) 계층이 있습니다. 총 6개 도구를 정의했습니다. `compute_ibo`는 분자 좌표를 받아서 IBO 계산과 시각화를 한방에 해줍니다. `visualize_orbital`은 이미 계산된 데이터를 3D로 렌더링합니다. `parse_output`은 양자화학 출력 파일을 분석합니다. `compute_partial_charges`는 IAO 기반 부분 전하를 계산합니다. `convert_format`은 구조 파일 포맷을 변환합니다. `analyze_bonding`은 결합 분석 결과를 텍스트로 반환합니다. 이 도구들이 FastMCP 3.x 프레임워크의 `@mcp.tool()` 데코레이터를 통해 MCP 서버에 등록되어, AI 에이전트가 프로토콜을 통해 호출할 수 있게 됩니다.

맨 위에 `mcp_server.py`가 있어서 FastMCP 서버 인스턴스를 생성하고 6개 도구를 묶어 외부에 노출합니다. `claude_desktop_config.json`을 통해 Claude Desktop과 바로 연결할 수 있는 설정 예시도 제공합니다.

## 오비탈 사진 시점까지 어디까지 왔었나

오비탈 이미지를 첨부한 시점, 그러니까 Phase β 실행 직전까지의 상태를 먼저 정리하겠습니다. 그때까지 완성된 것은 프로젝트 기반 구조 전체, 4개 백엔드의 핵심 로직, MCP 서버와 6개 도구 등록, 그리고 13개 자동화 테스트 전체 통과였습니다.

하지만 결정적으로 빠진 것이 있었습니다. `generate_cube`라는 메서드가 `NotImplementedError`를 던지고 있었고, 따라서 IBO를 계산하더라도 그걸 3D 이미지로 만들 수가 없었습니다. 계산 엔진은 있는데 렌더링 파이프라인이 끊겨 있는 상태였던 겁니다. 또한 예제 스크립트가 미검증 상태였고, GitHub 레포가 아직 생성되지 않았으며, 작성자 정보가 placeholder로 남아 있었습니다.

개발 과정에서 해결한 크리티컬 이슈도 다섯 건 있었습니다. FastMCP가 개발 도중에 3.0으로 메이저 업데이트되어서 버전 범위를 재조정해야 했고, JOSS 저널이 6개월 공개 이력을 요구하는 새 정책을 도입해서 원래 3~4주 안에 제출하려던 계획이 틀어졌고, PySCF가 Windows에서 안 돌아가서 CI를 플랫폼별로 분리해야 했고, PySCF가 표준 XYZ 파일 포맷을 못 읽어서 자동 변환 헬퍼를 만들어야 했고, IAO 부분 전하의 합이 0이 안 나오는 수치 안정성 문제를 Löwdin 정규직교화로 해결해야 했습니다.

## 오비탈 사진 이후 어디까지 진행했나

여기서부터가 오늘 새로 한 작업입니다. Phase β와 Phase γ, 두 단계를 실행했습니다.

### Phase β-1: E2E 시각화 파이프라인 완성

가장 큰 변화입니다. `generate_cube`가 드디어 구현되었습니다. PySCF의 `cubegen.orbital` 함수를 호출해서 임시 파일에 Gaussian cube 포맷 데이터를 쓰고, 그걸 문자열로 읽어서 반환합니다. 임시 파일은 항상 정리됩니다.

`viz_backend.py`에 `render_orbital_from_cube`라는 새 메서드가 추가되었습니다. cube 텍스트와 분자 좌표를 받아서 py3Dmol로 양의 등치면(파란색)과 음의 등치면(빨간색)을 동시에 그린 인터랙티브 HTML을 생성합니다.

`tools/core.py`의 `compute_ibo` 도구가 확장되어서, 이제 단순히 텍스트가 아니라 IBO 개수, SCF 에너지, IAO 전하 배열, 그리고 완성된 HTML 시각화까지 포함한 JSON dict를 반환합니다. 이로써 "분자 좌표를 넣으면 3D 오비탈 그림이 나온다"는 전체 파이프라인이 끊김 없이 연결되었습니다.

신규 테스트 3개가 추가되어 총 16개 테스트가 전부 통과했습니다.

### Phase β-2: 예제 스크립트 실동작 검증

`examples/01_water_ibo.py`를 WSL2에서 실행했을 때 실제 결과가 나왔습니다. 물 분자의 SCF 에너지가 -74.963023 Hartree로 수렴했고, IAO 부분 전하가 산소 -0.4706, 수소 각각 +0.2353으로 합계 정확히 0.000000이 나왔습니다. IBO는 5개가 생성되었는데, 물 분자의 전자 5쌍(O-H 결합 2개, O 고립전자쌍 2개, O 내각 전자 1개)에 정확히 대응합니다. cube 데이터는 40×40×40 격자로 843KB, 최종 HTML 파일은 1.7MB였습니다. 브라우저에서 열면 물 분자 위에 IBO 등치면이 3D로 회전 가능하게 표시됩니다.

에탄올 예제(`02_orca_parse_viz.py`)도 ASE 포맷 변환과 py3Dmol 시각화가 동작하도록 재작성했습니다.

### Phase β-3: GitHub 공개 릴리스

`.gitignore`를 작성하고, `CHANGELOG.md`에 v0.1.0-alpha 항목을 기록하고, README에 conda를 통한 설치 방법을 추가한 뒤, `https://github.com/dydtkddl/qcviz-mcp`에 public 레포로 push했습니다. JOSS가 6개월 공개 이력을 요구하므로, 이 시점(2026-03-05)부터 시계가 돌아가기 시작한 것입니다. 최소 2026년 9월 5일 이후에 제출 가능합니다.

push 직후에 `pyproject.toml`의 저자 정보가 `User <user@example.com>` placeholder 상태로 올라간 것을 발견하고, `Yongsang An <yongsang.an@khu.ac.kr>`로 수정했습니다. 프로젝트 URL도 `github.com/user/qcviz-mcp`에서 `github.com/dydtkddl/qcviz-mcp`로 고쳤습니다. `paper/paper.md`의 저자와 소속도 Yongsang An / Kyung Hee University로 업데이트했습니다.

### Phase γ-1: MCP 서버 라이브 검증

`tests/test_mcp_server.py`를 새로 만들어서 4가지를 검증했습니다. 서버 import가 되는지, 서버 이름이 "QCViz-MCP"인지, 6개 도구 함수가 core 모듈에 존재하는지, 그리고 가장 중요한 것으로 FastMCP 3.x의 `Client` 클래스를 통해 실제 MCP 프로토콜 수준에서 `list_tools()`를 호출했을 때 6개 도구가 반환되는지까지 확인했습니다. 단순히 Python import가 되는지가 아니라, MCP 프로토콜 레벨에서 서버가 제대로 응답하는지를 검증한 것입니다.

### Phase γ-2: 벤치마크 스위트 구축

`benchmark/molecules.py`에 NIST 표준 결합 길이를 기반으로 10개 분자를 정의했습니다. 물, 메탄, 에틸렌, 포름알데히드, 플루오린화수소, 암모니아, 일산화탄소, 질소, 시안화수소, 수소화리튬입니다. 각 분자마다 기대되는 IBO 개수를 미리 지정해 두었는데, 예를 들어 물은 5개, 에틸렌은 8개(C=C 이중결합 포함), 질소는 7개(삼중결합 포함) 같은 식입니다.

`benchmark/run_benchmark.py`를 만들어서 `python -m benchmark.run_benchmark`으로 실행하면 10개 분자 전부에 대해 SCF→IAO→IBO를 돌리고, IBO 개수가 기대값과 맞는지, 전하 합이 0인지를 확인하는 표를 출력합니다. 실행 결과 10개 분자 전부 통과했습니다. 모든 분자에서 IBO 개수가 기대값과 정확히 일치하고, 전하 합이 -0.000000 또는 0.000000으로 나왔습니다.

`tests/test_benchmark.py`에는 20개 파라미터화된 테스트(10개 분자 × IBO 개수 검증 + 10개 분자 × 전하 보존 검증)를 추가했습니다.

### Phase γ-3: 테스트 커버리지 확장

기존 테스트에 추가로 여러 건을 작성했습니다. `test_pyscf_backend.py`에 `_parse_atom_spec` 헬퍼의 정상/비정상 입력 테스트 2개, DFT(B3LYP) 계산 테스트, Boys 국소화 테스트를 추가했습니다. `test_ase_backend.py`에 XYZ→CIF 왕복 변환 테스트와 존재하지 않는 파일 에러 테스트를 추가했습니다. `test_tools.py`에 `compute_partial_charges`, `convert_format`, `analyze_bonding` 도구의 테스트를 추가했습니다.

Windows에서는 15개 통과, 8개 스킵(PySCF 및 벤치마크 관련). Linux(WSL2)에서는 벤치마크 테스트의 import 에러 한 건을 제외하면 전부 통과입니다.

### Phase γ-4: 릴리스 준비

`CHANGELOG.md`를 Phase γ 내용으로 업데이트하고, `benchmark/results/`를 `.gitignore`에 추가하고, `pytest-asyncio`를 의존성에 추가하고, `asyncio_mode = "auto"`를 pytest 설정에 넣었습니다. 그리고 GitHub에 push까지 완료했습니다.

## 현재 남아있는 잔여 이슈 두 건

첫째, `tests/test_benchmark.py`에서 `from benchmark.molecules import MOLECULES`가 `ModuleNotFoundError`를 일으킵니다. `benchmark/` 디렉토리가 `src/` 바깥에 있어서 pytest가 자동으로 import 경로에 포함시키지 못하기 때문입니다. `pyproject.toml`의 pytest 설정에 `pythonpath = ["."]` 한 줄을 추가하면 프로젝트 루트가 import 경로에 들어가서 해결됩니다.

둘째, `git tag -a v0.1.0-alpha`가 "tag already exists" 에러를 냅니다. 이전에 이미 같은 이름의 태그를 만들었기 때문입니다. 기존 태그를 삭제하고 다시 만들거나, `v0.2.0-alpha`처럼 새 버전으로 태그를 만들면 됩니다. JOSS 이력 축적 측면에서는 릴리스가 여러 개 보이는 게 유리하므로 새 버전 태그를 추천합니다.

## 숫자로 보는 현재 상태

테스트는 Linux 기준으로 벤치마크 import 이슈만 해결하면 약 40개 이상이 통과할 것으로 예상됩니다. 벤치마크 실행은 10개 분자 전부 성공했습니다. E2E 파이프라인은 분자 좌표 입력부터 3D HTML 출력까지 완전히 동작합니다. GitHub 레포는 공개 상태이고 커밋 이력이 쌓이고 있습니다.

개발 진척률은 약 75%입니다. 오비탈 사진 시점의 70%에서 벤치마크 스위트와 MCP 프로토콜 검증이 추가되어 올라갔습니다.

JOSS 투고 진척률은 약 15%입니다. 코드 완성도가 올라간 것이 반영되었지만, 6개월 공개 이력 요건이 여전히 가장 큰 블로커입니다.

## 앞으로 해야 할 일

### 즉시 해야 할 것 (75% → 80%)

위에서 말한 잔여 이슈 두 건을 해결해야 합니다. `pyproject.toml`에 `pythonpath` 설정을 추가하고, 새 버전 태그를 만들어서 push합니다. 그리고 GitHub에 이슈 5개를 등록해야 합니다. 기저함수 벤치마크 확장(cc-pVDZ, cc-pVTZ), IboView 브릿지(molden 파일 내보내기), 헤드리스 PNG 내보내기(Playwright 활용), API 문서 완성, UHF 개각 시스템 지원. 이 이슈들은 코드를 당장 짜지 않더라도 JOSS가 요구하는 "공개 개발 활동"의 증거가 됩니다.

### 단기적으로 해야 할 것 (80% → 90%)

테스트 커버리지를 `pytest --cov`로 측정해서 75% 이상으로 끌어올려야 합니다. `ruff check`과 `ruff format`으로 전체 코드의 린트 에러를 0건으로 만들어야 합니다. 나머지 4개 도구(`parse_output`, `compute_partial_charges`, `convert_format`, `analyze_bonding`)가 실제로 의미 있는 결과를 반환하는지 검증해야 합니다. 현재 등록은 되어 있지만 일부는 스텁에 가까울 수 있습니다. API 문서를 mkdocs로 작성하고, 사용 가이드를 보강해야 합니다. 벤치마크를 sto-3g 외에 cc-pVDZ 같은 더 큰 기저함수로도 돌려봐야 합니다.

### 중기적으로 해야 할 것 (90% → 투고 준비)

3월부터 8월까지 6개월 동안 GitHub에 꾸준히 활동을 남겨야 합니다. 이슈를 열고, PR을 만들고, 버전을 올리고, 릴리스를 찍는 겁니다. v0.2.0에서 벤치마크 확장, v0.3.0에서 IboView 브릿지, v0.4.0에서 문서 완성, v1.0.0에서 JOSS 제출용 안정 릴리스 같은 로드맵을 잡으면 됩니다.

`paper/paper.md`의 본문을 실질적으로 작성해야 합니다. 현재는 섹션 제목과 골격만 있는 상태입니다. Summary, Statement of Need, State of the Field, Software Design, Research Impact Statement, AI Usage Disclosure 각 섹션에 실제 내용을 채워야 합니다.

아키텍처 다이어그램과 예제 출력 스크린샷을 figure로 만들어야 합니다. JOSS 논문에 figure가 필수는 아니지만, 리뷰어 인상에 큰 차이를 만듭니다.

`paper.bib`를 현재 6개에서 최소 10개 이상으로 확장해야 합니다. MCP 프로토콜 관련 논문, 유사 프로젝트 논문, 양자화학 교육 관련 논문 등을 추가합니다.

Zenodo에 레포를 아카이빙해서 DOI를 발급받아야 합니다. JOSS 제출 시 필수입니다.

ORCID가 없다면 만들어야 합니다. JOSS 논문 저자에 ORCID가 필수입니다.

가능하다면 외부 사용자(연구실 동료, 다른 대학 계산화학 연구자)에게 써보게 하고 피드백을 받는 것이 좋습니다. JOSS 리뷰에서 "커뮤니티 참여" 항목이 있기 때문입니다.

### 논문 병행 준비 (JCIM)

JOSS 대기 기간 동안 JCIM(Journal of Chemical Information and Modeling) 벤치마크 논문을 병행 준비할 수 있습니다. 15~20개 분자에 대해 PySCF IBO와 IboView IBO의 수치를 비교하고, LLM 워크플로우에서의 사용성 가속 효과를 정량적으로 보여주는 논문입니다. JOSS와 스코프가 겹치지 않도록 JOSS는 소프트웨어 설명, JCIM은 과학적 벤치마크에 각각 집중하면 자기표절 문제가 없습니다.
