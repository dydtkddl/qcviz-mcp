## 섹션 1: 2줄 요약

**QCViz-MCP는 IAO/IBO 오비탈 로컬라이제이션과 3D 시각화를 MCP(Model Context Protocol) 프로토콜로 제공하는, 2026년 3월 현재 세계 유일의 전자구조 수준 양자화학 MCP 서버이다.**

**이 시스템은 "물 분자의 결합 오비탈을 보여줘"라는 자연어 한 줄로, 기존에 PySCF 스크립트 작성→SCF 수행→IAO/IBO 로컬라이제이션→cube 파일 생성→IboView 실행이라는 5단계 전문가 워크플로우를 1단계로 축약함으로써, 양자화학 오비탈 분석의 접근성을 근본적으로 변화시킨다.**

## 섹션 2: 시스템 명세서

### 2.1 아키텍처 개요

QCViz-MCP는 3계층 구조로 설계되었다. 최상위에는 MCP 프로토콜 층이 있으며, Anthropic이 2024년 11월 25일에 공개한 Model Context Protocol 사양을 따른다. 이 층은 FastMCP 3.x 프레임워크(2026년 2월 18일 GA 릴리스, 현재 v3.0.2)를 기반으로 하며, async-native 설계와 SSE(Server-Sent Events) transport를 채택했다. 중간에는 도구 층이 위치하고, 6개의 MCP 도구가 등록되어 있다. 최하위에는 백엔드 층이 있으며, 4개의 계산·시각화·파싱 백엔드가 BackendRegistry를 통해 자동 등록된다. BackendRegistry는 플러그인 아키텍처를 구현한 것으로, 새로운 백엔드(예: Psi4, ORCA 직접 실행)를 추가할 때 기존 코드를 수정하지 않고 등록만 하면 된다.

### 2.2 MCP 도구 6개

시스템은 6개의 MCP 도구를 제공하며, 각각의 기능과 성숙도는 다음과 같다.

| 도구                      | 기능                                                                                | 핵심 의존성 | 성숙도    |
| ------------------------- | ----------------------------------------------------------------------------------- | ----------- | --------- |
| `compute_ibo`             | SCF→IAO→IBO 전체 파이프라인 실행, cube 파일 생성(80³ 그리드), HTML 3D 시각화 반환   | PySCF       | 논문 수준 |
| `visualize_orbital`       | cube 파일을 py3Dmol 기반 인터랙티브 3D HTML로 렌더링                                | py3Dmol     | 논문 수준 |
| `parse_output`            | ORCA, Gaussian 16, PySCF 출력 파일을 파싱하여 에너지·좌표·전하·오비탈 에너지를 추출 | cclib 1.8.1 | 기능 완료 |
| `compute_partial_charges` | Mulliken, Löwdin, IAO 세 가지 방식으로 부분 원자 전하 계산                          | PySCF       | 기능 완료 |
| `convert_format`          | 분자 구조 파일 포맷 변환 (xyz↔cif↔pdb↔POSCAR↔extxyz)                                | ASE         | 기능 완료 |
| `analyze_bonding`         | IBO 기반 원자 기여도 분석, 각 결합 오비탈에 대한 원자별 기여 비율 산출              | PySCF       | 기능 완료 |

"논문 수준"이란 벤치마크 검증 완료, 엣지 케이스 테스트 통과, 보안 검증까지 마친 상태를 의미한다. "기능 완료"는 핵심 기능이 동작하고 테스트를 통과하지만, 대규모 분자나 특이 조건에서의 추가 검증이 필요한 상태를 의미한다.

### 2.3 백엔드 4개

**PySCFBackend**은 시스템의 핵심 계산 엔진이다. RHF/UHF 수준의 SCF(Self-Consistent Field) 계산을 수행하고, Gerald Knizia가 2013년 JCTC에서 제안한 IAO(Intrinsic Atomic Orbitals)와 IBO(Intrinsic Bond Orbitals) 로컬라이제이션을 실행한다. cube 파일 생성(기본 80³ 그리드, 출판 수준), Molden 포맷 내보내기(IBO/IAO/canonical 오비탈), Mulliken/Löwdin/IAO 부분 전하 계산을 담당한다.

**CclibBackend**은 cclib 1.8.1 라이브러리를 통해 ORCA, Gaussian 16, PySCF 등 다수 양자화학 프로그램의 출력 파일을 파싱한다. SCF 에너지, 원자 좌표, Mulliken 전하, 오비탈 에너지(HOMO/LUMO), 기저함수 수 등을 표준화된 형식으로 추출한다.

**Py3DmolBackend**은 py3Dmol 라이브러리를 사용하여 cube 파일을 인터랙티브 HTML 3D 시각화로 변환한다. 사용자는 웹 브라우저에서 오비탈을 회전·확대하며 관찰할 수 있다.

**ASEBackend**은 ASE(Atomic Simulation Environment)를 사용하여 분자 구조 파일의 포맷 변환을 처리한다. xyz, cif, pdb, POSCAR, extxyz 등 다양한 형식 간의 양방향 변환을 지원한다.

### 2.4 파이프라인 흐름

사용자가 Claude Desktop에서 "물 분자의 IBO를 계산하고 시각화해줘"라고 입력하면, 다음과 같은 흐름이 실행된다.

```
사용자 자연어 입력
  → Claude Desktop (LLM 해석)
    → MCP 프로토콜 (JSON-RPC over SSE)
      → compute_ibo 도구 호출 (atom_spec="O 0 0 0; H 0 0.757 0.587; H 0 -0.757 0.587")
        → PySCFBackend.compute_scf()    : RHF/STO-3G, SCF 에너지 = -74.963 Hartree
        → PySCFBackend.compute_iao()    : 7개 IAO 생성
        → PySCFBackend.compute_ibo()    : 5개 IBO (2 O-H 결합 + 2 lone pair + 1 core)
        → PySCFBackend.generate_cube()  : 80×80×80 그리드, ~843 KB/orbital
        → Py3DmolBackend.render()       : 인터랙티브 HTML (~1.7 MB)
      → JSON 응답 반환
    → MCP 프로토콜
  → Claude Desktop
사용자에게 결과 표시 (에너지, IBO 수, 전하 배열, 3D 뷰어)
```

전체 파이프라인의 실행 시간은 water/STO-3G 기준 약 0.1초, ethylene/cc-pVDZ 기준 약 0.8초이다.

### 2.5 정량 지표

테스트는 총 95개가 통과하며 실패가 0건이다. 실행 시간은 14.46초(Linux/WSL2)이다. 테스트 구성은 보안 테스트 6개, 백엔드 단위 테스트 약 30개, MCP 서버 통합 테스트 약 10개, 벤치마크 파라미터화 테스트 40개(10분자 × 2기저 × 2항목), 파싱/전하/변환 테스트 약 20개, IBO 검증 테스트 약 6개로 이루어져 있다.

벤치마크는 10개 NIST 표준 분자(H₂O, CH₄, NH₃, HF, C₂H₄, CH₂O, C₂H₂, CH₃OH, N₂, CO)에 대해 STO-3G와 cc-pVDZ 두 가지 기저함수로 실행하여 총 20건이 전부 성공했다. IBO 카운트는 두 기저함수에서 동일하게 유지되어 기저함수 독립성이 확인되었고, 총 Mulliken 전하는 모든 분자에서 0.000000으로 전하 보존이 입증되었다.

보안 검증으로는 파일 경로 순회 공격 차단(`_validate_file_path`), 과도한 원자 수 입력 제한(`_validate_atom_spec`, 기본 50개), 입력 크기 검사를 구현했다. 이는 MCP 보안 모범사례(TheNewStack, 2025-09-15)와 CVE-2026-0755(도구 중독 취약점) 대응을 반영한 것이다.

IBO 품질 검증 파이프라인은 세 가지 지표를 측정한다. 오비탈 spread(σ², Å² 단위)로 로컬라이제이션 정도를 정량화하고, Molden roundtrip 검증(export→re-import→Frobenius norm)으로 데이터 무결성을 확인하며, IBO 기반 전하와 Mulliken 전하 간의 부호 일치율로 화학적 일관성을 검증한다.

### 2.6 의존성 및 환경

실행 환경은 Python 3.10 이상이며, PySCF는 conda-forge 채널에서 설치한다(Linux/macOS 전용, Windows는 WSL2 필요). cclib 1.8.1, py3Dmol(최신 릴리스 2026-01-22, 단 원본 repo avirshup/py3dmol은 유지보수 중단 상태), ASE, FastMCP 3.0.2+가 필요하다. 개발 의존성으로는 pytest, pytest-asyncio, ruff, mkdocs-material이 있다. 라이선스는 BSD-3-Clause이며, ASE 컴포넌트는 LGPL로 QCViz-MCP가 ASE를 import하여 사용하는 형태이므로 라이선스 호환성에 문제가 없다.

## 섹션 3: 노벨티

### 3.1 노벨티 ①: 도메인 레벨 — 전자구조 양자화학의 MCP 진입

QCViz-MCP의 첫 번째이자 가장 근본적인 새로움은, 전자구조 수준의 양자화학 계산을 MCP 프로토콜로 노출한다는 점이다. 여기서 "전자구조 수준"이란 분자의 전자 파동함수를 직접 다루는 계산을 의미한다. 구체적으로, Knizia(2013, JCTC, 801+ citations)가 제안한 IAO/IBO 로컬라이제이션을 통해 분자의 결합 구조를 화학적으로 직관적인 오비탈로 분해하고, 이를 3차원으로 시각화하는 전체 과정을 하나의 MCP 도구 호출로 수행한다.

2026년 3월 기준, 화학 분야의 MCP 서버는 네 가지가 확인된다. ChemLint(molML 그룹, ChemRxiv 2026-02-24)는 분자 기계학습 워크벤치로서 분자 기술자(descriptor) 계산과 SMILES 기반 속성 예측에 강점이 있으나, 전자구조 계산이나 오비탈 로컬라이제이션 기능은 없다. Argonne 국립연구소의 mcp.science(arXiv:2508.18489, 2025-08)는 Globus Compute를 통한 원격 HPC 작업 실행 인프라로서, 범용적이고 확장성이 뛰어나지만, 도메인 특화 로직(IAO/IBO 파이프라인 등)을 내장하지 않으며, 사용자가 직접 PySCF 스크립트를 작성해야 한다. DataWarrior MCP(ChemRxiv 2026-02-05)는 케모인포매틱스 도구의 자연어 제어에 초점을 맞추며, 양자역학 계산은 범위 밖이다. El Agente Q(Matter, 2025-07-02, Cell Press)는 LLM 기반 자율 에이전트로서 양자화학 워크플로우를 자동 생성하는 인상적인 시스템이나, MCP 프로토콜이 아닌 커스텀 에이전트 아키텍처를 사용하므로 Claude Desktop 등 MCP 호환 클라이언트와의 상호운용성이 없다.

이들 중 어느 것도 "IAO/IBO 로컬라이제이션 + 3D 시각화 + MCP 프로토콜"의 조합을 제공하지 않는다.

### 3.2 노벨티 ②: 인터페이스 레벨 — 5단계를 1단계로

기존에 연구자가 물 분자의 IBO를 시각화하려면 다음 5단계를 거쳐야 했다. (1) PySCF 스크립트 작성(분자 정의, 기저함수 선택, SCF 실행), (2) IAO 생성 코드 추가, (3) IBO 로컬라이제이션 코드 추가, (4) cube 파일 생성(cubegen 호출), (5) IboView 또는 VMD를 열어 시각화. 이 과정은 최소 20줄 이상의 Python 코드와 두 개 이상의 프로그램을 요구하며, PySCF API에 대한 상세한 지식이 전제된다.

QCViz-MCP는 이 전체를 "compute_ibo water" 수준의 자연어 한 줄로 축약한다. LLM이 사용자 의도를 해석하고, MCP 프로토콜을 통해 적절한 도구를 호출하며, 결과를 JSON으로 반환한다. 사용자는 PySCF API를 알 필요가 없고, cube 파일의 존재를 인지할 필요도 없다. 이는 양자화학 오비탈 분석의 진입 장벽을 "Python 프로그래밍 + 양자화학 이론"에서 "자연어 설명 능력"으로 낮춘다.

### 3.3 노벨티 ③: 검증 레벨 — 내장된 품질 보증

QCViz-MCP는 단순히 계산을 수행하는 것을 넘어, 계산 결과의 품질을 자체적으로 검증하는 파이프라인을 내장한다. 이는 세 가지 축으로 구성된다. 오비탈 spread(σ²)를 측정하여 IBO가 진정으로 로컬라이즈되었는지 정량적으로 확인하고, 로컬라이제이션 그래디언트 노름이 1e-8 이하인지 검사하여 안장점이 아닌 극소점에 수렴했음을 보장하며, Molden roundtrip(export→re-import)의 Frobenius 노름이 사실상 0인지 확인하여 데이터 무결성을 입증한다.

이러한 내장 검증 파이프라인을 MCP 도구와 함께 제공하는 것은 기존의 어떤 시스템에서도 찾아볼 수 없다. IboView는 GUI에서 오비탈을 시각적으로 확인할 수 있지만 프로그래밍 가능한 품질 지표를 제공하지 않으며, PySCF 자체에도 IBO 품질을 자동으로 검증하는 유틸리티는 포함되어 있지 않다.

## 섹션 4: 논문이 되는 이유

### 4.1 Substantial Scholarly Effort

이 소프트웨어의 개발에는 양자화학 도메인 지식과 소프트웨어 엔지니어링이 결합되었다. 도메인 측면에서는 IAO/IBO 이론의 구현 세부사항(MINAO 기저, Pipek-Mezey 로컬라이제이션의 수렴 조건, 안장점 문제), SCF 수렴 전략, 기저함수 선택의 영향(STO-3G vs cc-pVDZ에서 IBO 카운트의 불변성), 그리고 10개 벤치마크 분자의 선정과 기대값 설정에 전문 지식이 투입되었다. 엔지니어링 측면에서는 MCP 프로토콜 보안 설계(CVE-2026-0755 대응, 경로 검증, 입력 제한), 플러그인 아키텍처(BackendRegistry), async-native 도구 등록, cube 해상도 트레이드오프(40³→80³로 상향 시 파일 크기 8배 증가 대비 시각적 품질 향상), 그리고 5단계 체계적 개발 프로세스(Phase α→β→γ→δ→ε)가 적용되었다.

95개 테스트, 20개 벤치마크, IBO 검증 파이프라인, 6개 MCP 도구, 4개 백엔드, MkDocs API 문서, Claude Desktop 통합 가이드를 포함하는 이 시스템은 JOSS가 2020년에 제시한 "약 3개월의 개발자 시간"에 해당하는 실질적 노력을 반영한다.

### 4.2 Research Purpose

양자화학에서 오비탈 분석은 화학 결합의 본질을 이해하는 핵심 도구이다. 그러나 현재의 오비탈 분석 워크플로우는 높은 진입 장벽을 가진다. 연구자는 PySCF나 Psi4의 API를 숙지해야 하고, cube 파일을 수동으로 생성해야 하며, IboView나 VMD 같은 별도의 시각화 프로그램을 설치하고 운용해야 한다. 이 파편화된 워크플로우는 비전문가(학부생, 타 분야 연구자)의 접근을 사실상 차단한다.

QCViz-MCP는 MCP 프로토콜과 LLM의 결합을 통해 이 문제를 해결한다. "에틸렌의 파이 결합을 보여줘"라는 자연어 한 줄로 전체 파이프라인이 실행되므로, 프로그래밍 지식 없이도 오비탈 분석이 가능하다. 이는 양자화학 교육(학생이 결합 이론을 시각적으로 학습)과 연구(빠른 탐색적 분석)에 직접적인 가치를 제공한다.

### 4.3 State of the Field 대응

섹션 3의 비교 테이블이 이를 담당한다. 핵심 논점은 "build vs contribute"이다. PySCF는 양자화학 계산 엔진이고, IboView는 오비탈 시각화 GUI이다. QCViz-MCP는 이들의 기능을 복제하는 것이 아니라, MCP 프로토콜을 통해 LLM과 연결하는 새로운 계층을 구축한 것이다. 이는 PySCF에 코드를 기여하거나 IboView에 MCP 플러그인을 추가하는 것과는 근본적으로 다른 아키텍처적 결정이며, 6개 도구의 통합된 인터페이스, 내장 벤치마크, 품질 검증이라는 독자적 가치를 창출한다.

### 4.4 Software Design 대응

설계 결정의 핵심은 세 가지이다. 첫째, FastMCP 3.x를 선택한 이유는 async-native 설계로 장시간 SCF 계산 중에도 서버가 블로킹되지 않으며, 데코레이터 기반 도구 등록(`@mcp.tool()`)으로 새 도구 추가가 간결하고, SSE transport로 Claude Desktop과의 통합이 즉시 가능하기 때문이다. 둘째, BackendRegistry 플러그인 패턴을 도입하여 새 백엔드(예: GPU4PySCF, Psi4) 추가 시 기존 코드 수정 없이 등록만 하면 되도록 했다. 셋째, MCP 보안 모범사례를 적극 적용하여 경로 순회 공격, 과도한 입력, 도구 중독 공격에 대비했다.

### 4.5 Research Impact 대응

현재 시점에서 외부 사용자는 0명이며, 이는 솔직히 인정해야 할 한계이다. JOSS는 "evidence of realized impact or credible near-term significance"를 요구하므로, 후자에 집중한다. 20개 벤치마크(10분자 × 2기저)가 100% 성공하여 IBO 카운트의 화학적 정확성과 기저함수 독립성을 정량적으로 입증했다. IBO 품질 검증 파이프라인(spread, gradient, roundtrip)은 다른 IBO 구현의 참조 검증 도구로 활용될 수 있다. Claude Desktop 통합 가이드(`claude_desktop_config.json`)를 제공하여 설치 후 즉시 사용 가능하다.

### 4.6 AI Usage Disclosure 대응

JOSS는 2026년 1월 "Preparing JOSS for a generative AI future" 블로그 포스트에서 AI 보조 개발에 대한 새 기준을 발표했다. 핵심은 코드 줄 수가 아닌 인간의 실질적 기여(설계, 도메인 추상화, 테스트 설계, 문서화)를 평가한다는 것이다. QCViz-MCP의 AI 사용 공개는 이에 맞춰 작성되었다. Claude(Anthropic)를 아키텍처 토론, 코드 스캐폴딩, 테스트 생성 보조에 사용했으며, 모든 양자화학 도메인 로직, 벤치마크 분자 선정, 보안 설계, 코드 리뷰는 인간 저자가 수행했다. AI 생성 코드는 95개 자동화 테스트와 20개 벤치마크를 통과한 경우에만 반영되었다.

### 4.7 현재 한계에 대한 자기비판

투명성을 위해 QCViz-MCP의 현재 한계를 명시한다.

첫째, 외부 사용자가 0명이다. JOSS의 "collaborative effort" 요건을 충족하려면 최소 1명 이상의 외부 기여자 또는 사용자 피드백이 필요하다. 이는 6개월간의 GitHub 공개 활동을 통해 확보해야 한다.

둘째, 전이금속을 지원하지 않는다. PySCF의 IAO 구현은 ECP(Effective Core Potential)를 사용하는 전이금속에서 MINAO 기저가 부적합하다는 경고를 발생시킨다. 4주기 이상의 원소에 대한 IBO 로컬라이제이션은 검증되지 않았다.

셋째, py3Dmol의 원본 저장소(avirshup/py3dmol)가 유지보수 중단 상태이다. 기능적으로는 문제가 없으나, 장기적으로 대안(3Dmol.js 직접 통합 등)을 검토해야 한다.

넷째, 헤드리스 PNG 내보내기가 미구현이다. 현재 3D 시각화는 HTML만 제공하며, 논문용 정적 이미지를 얻으려면 스크린샷을 수동으로 캡처하거나 Playwright 기반 파이프라인을 별도로 구축해야 한다.

다섯째, UHF/ROHF 지원이 실험적이다. O₂ 삼중항에 대한 UHF 분기 코드는 존재하나, 대규모 오픈셸 시스템에서의 검증은 불충분하다.

## 섹션 5: 논문 서사 흐름

### 5.1 서사 설계

JOSS paper.md는 "문제 → 해법 → 증거 → 의의"의 4막 구조를 따른다. 750-1750 단어 제한 내에서, 각 섹션은 다음과 같은 핵심 메시지를 전달하고 다음 섹션으로의 논리적 연결 고리를 형성한다.

**제1막: Summary (도입)**

핵심 메시지는 "QCViz-MCP는 IAO/IBO 오비탈 로컬라이제이션과 3D 시각화를 자연어 인터페이스로 제공하는 MCP 서버"이다. 비전문가 독자를 위해 MCP와 IBO를 각각 1문장으로 설명한다. MCP는 "AI 모델이 외부 도구를 호출할 수 있게 하는 표준 프로토콜"로, IBO는 "분자의 전자 구조를 화학자가 직관적으로 이해할 수 있는 결합 오비탈로 분해하는 방법"으로 소개한다.

**제2막: Statement of Need → State of the Field (문제 정의)**

Statement of Need에서는 문제를 세 층으로 제시한다. (1) 양자화학 오비탈 분석은 연구와 교육 모두에서 핵심이다. (2) 그러나 현재 워크플로우는 5단계의 수동 과정을 요구하며, PySCF API 지식이 전제된다. (3) MCP와 LLM의 등장으로 이 장벽을 자연어 인터페이스로 해소할 기회가 열렸다.

State of the Field에서는 비교 테이블을 제시하고 "build vs contribute" 논점을 다룬다. PySCF는 계산 엔진이고 IboView는 시각화 GUI이므로, MCP 프로토콜을 통한 통합 인터페이스는 기존 도구에 대한 기여(contribute)가 아니라 새로운 계층의 구축(build)이라는 점을 명확히 한다. ChemLint, mcp.science, El Agente의 강점을 인정하면서도, "전자구조 + MCP + 시각화 + 검증"의 조합이 부재함을 보인다.

**제3막: Software Design → Research Impact (해법과 증거)**

Software Design에서는 3계층 아키텍처, FastMCP 3.x 선택 이유, BackendRegistry 플러그인 패턴, 보안 설계의 3가지 핵심 결정을 기술한다. 파이프라인 흐름도(Figure 1)를 통해 자연어 입력에서 JSON 결과까지의 전체 경로를 시각적으로 보여준다.

Research Impact에서는 정량적 증거를 제시한다. 벤치마크 결과 테이블(Table 1: 10분자 × 2기저, 에너지/IBO수/전하/시간)과 경쟁 도구 비교 테이블(Table 2: 7개 능력 × 6개 도구)이 핵심 데이터이다. "자연어 한 줄 vs 기존 5단계" 비교(Figure 2)로 접근성 개선을 시각화한다. 외부 사용자가 아직 없다는 한계를 솔직히 밝히되, 벤치마크의 재현 가능성과 Claude Desktop 즉시 사용 가능성을 "credible near-term significance"로 제시한다.

**제4막: AI Usage Disclosure (투명성)**

AI 사용을 개발·문서·논문 세 영역으로 분리하여 공개하고, 각 영역에서 AI의 역할과 인간의 역할을 명확히 구분한다. 검증 방법(95개 테스트, 20개 벤치마크, 수동 코드 리뷰)을 구체적으로 제시한다.

### 5.3 참고문헌 구성 (최소 12개)

핵심 이론 문헌으로 Knizia의 2013년 JCTC 논문(IAO/IBO 원론)이 있다. 소프트웨어 의존성으로는 PySCF(Sun et al., 2018, 2020), cclib(O'Boyle et al., 2008; Lam et al., 2024), py3Dmol(Rego & Koes, 2015), ASE(Larsen et al., 2017)가 포함된다. 프로토콜 관련으로 MCP 사양(Anthropic, 2024)과 FastMCP(Lowin, 2026)를 인용한다. 경쟁 도구 문헌으로 ChemLint(2026, ChemRxiv), mcp.science(Foster et al., 2025, arXiv), DataWarrior MCP(2026, ChemRxiv), El Agente(2025, Matter)를 포함한다. 보안 관련으로 MCP 보안 모범사례(TheNewStack, 2025)를 참조한다. 총 12개 이상의 참고문헌이 확보된다.
