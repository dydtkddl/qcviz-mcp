## Axis 2: PySCF IBO 한계

### 알려진 이슈 수집

**GitHub issue #1032 (IBO + virtual orbitals)**: IBO 국소화는 점유 궤도에만 적용 가능하다. 가상 궤도에 적용 시 결과가 나쁘다. 이는 알고리즘적 제한으로 Knizia의 원래 논문에서도 점유 궤도만 대상으로 한다. QCViz-MCP의 현재 구현은 점유 궤도만 처리하므로 이 제한에 해당하지 않으나, 문서에 명시적으로 기술해야 한다.

**GitHub discussions #3071 (안장점 수렴)**: PySCF의 궤도 국소화 옵티마이저가 안장점에 수렴하는 문제가 보고되었다. Boys 국소화에서 벤젠의 경우 확인됨 (issue #432). PM(Pipek-Mezey) 국소화에서도 발생 가능하나 상대적으로 드물다. 관련 논문에서 "poor convergence behavior i.e. convergence onto saddle points of the PySCF orbital optimizer"를 문서화하였다.

**discussions #3071 (성능)**: cc-pVQZ 기저로 메탄올의 PM 국소화에 3.4초가 소요되며, Psi4 대비 느리다는 보고. 큰 기저함수에서 국소화 속도가 병목이 될 수 있다.

### Edge cases 정리

| Edge case | 현재 상태 | 영향도 | 대응 방안 |
|---|---|---|---|
| **가상 궤도 국소화** | 미지원 (IBO 설계상 점유 궤도만) | LOW (설계 의도) | README/docs에 명시 |
| **UHF/ROHF 개각 시스템** | 미지원 (현재 RHF/RKS만) | HIGH (라디칼, 전이금속) | `compute_scf`에 UHF/ROHF 분기 추가 필요 |
| **전이금속 (4주기+)** | 미검증 | HIGH | IAO에서 `minao` 기저에 중원소가 포함되지 않을 수 있음; PySCF 경고: "ECP/PP is used. MINAO is not a good reference AO basis in IAO." |
| **큰 기저함수 (cc-pVQZ+)** | 미검증 | MEDIUM | 국소화 시간 증가, cubegen 메모리 증가 |
| **원자 50개 이상 분자** | 미검증 | MEDIUM | cubegen: 40³ grid × N_atoms AO → 메모리 O(N³), 80³에서 ~8배 증가 |
| **안장점 수렴** | 감지 메커니즘 없음 | HIGH | 국소화 후 gradient 크기를 확인하여 경고 반환 필요 |
| **PM vs Boys 선택** | 둘 다 지원하나 가이드 없음 | LOW | PM은 σ/π 분리 유지, Boys는 동가 곡선 결합(bent bonds) 생성; 기본값 PM 유지, 문서에 설명 추가 |

### Pipek-Mezey vs Boys 비교

Pipek-Mezey(PM)는 원자별 population의 제곱합을 최대화하여 σ/π 궤도의 분리를 유지한다. 에틸렌에서 PM은 C=C의 σ와 π를 별개 궤도로 보여준다. Boys는 궤도 간 거리 제곱합을 최대화하여 "곡선 결합(banana bonds)"을 생성한다. 에틸렌에서 Boys는 두 개의 동가 결합 궤도를 만든다. 화학적 직관과 일치하는 쪽은 PM이므로, IBO의 기본 국소화가 PM인 것은 적절하다. 다만 대칭성이 높은 분자(벤젠)에서 Boys가 안장점에 빠지기 쉽다는 보고가 있다.

### 핵심 질문 답변: 실제 연구자 워크플로우에서 깨지는 edge case는?

가장 위험한 것은 **(1) UHF/ROHF 미지원**(유기 라디칼, 전이금속 착물 사용 불가)과 **(2) 전이금속 + ECP에서 IAO MINAO 기저 경고**(결과의 화학적 의미가 떨어짐)이다. 안장점 수렴 문제는 드물지만 감지 없이 잘못된 결과를 반환할 위험이 있다.

## Axis 3: MCP 보안 + 프로덕션 준비도

### 2026년 MCP 보안 위협 현황

MCP의 첫해(2025-2026)에 30개 이상의 CVE가 보고되었다. 주요 위협 유형은 다음과 같다.

**Tool Poisoning (TPA)**: 악성 MCP 서버가 도구 설명(description)에 숨겨진 지시를 삽입하여 LLM을 조작. 성공률 72.8% (Repello AI 보고). QCViz-MCP는 자체 서버이므로 이 위협의 *소스*가 아니라 *대상*이 아님 — 다만 QCViz-MCP의 도구 description이 다른 악성 MCP 서버에 의해 shadowing될 수 있다.

**Command Injection (CVE-2026-0755)**: Gemini MCP tool에서 발견된 명령어 주입 취약점. 사용자 입력이 시스템 명령어에 직접 전달되는 경우.

**Prompt Injection via Tool Output**: 도구 출력에 포함된 악성 텍스트가 LLM의 다음 행동을 조작.

### QCViz-MCP 도구별 취약점 분석

### 즉시 필요한 보안 조치

1. **`parse_output`의 파일 경로 검증**: 허용된 디렉토리(예: `examples/`, `benchmark/`) 외부 경로를 거부. `os.path.realpath`로 심볼릭 링크 해제 후 접두사 확인.
2. **`convert_format`의 출력 경로 검증**: 쓰기 경로를 지정된 출력 디렉토리로 제한.
3. **입력 크기 제한**: `atom_spec`의 원자 수를 200개 이하로 제한, cube 데이터 크기 상한 설정.
4. **FastMCP 3.x `AuthMiddleware`**: 현재 미적용 상태. 로컬 서버에서는 필수 아니지만, `restrict_tag`로 도구 접근을 분류하는 것을 권장.

### 도구별 성숙도 평가

| **analyze_bonding** | ❌ 스텁 수준 의심 | ❌ | ❌ **불가** | 실질적 분석 로직의 깊이 확인 필요 |

### 일반화 부족 지점

**기저함수**: 벤치마크가 sto-3g에서만 수행됨. 연구 실무에서 sto-3g를 쓰는 경우는 극히 드물고, 최소 cc-pVDZ가 기본이다. cc-pVDZ/cc-pVTZ 벤치마크가 없으면 JOSS 리뷰어("이것이 toy example 이상인가?")의 질문에 답할 수 없다.

**전이금속 착물**: IAO의 `minao` 참조 기저에 전이금속이 포함되어 있긴 하나, ECP(유효 내각 전위)가 사용되면 PySCF가 경고를 발생시킨다. 전이금속 착물은 QCViz-MCP의 주요 타겟 사용 사례(유기금속 반응 메커니즘)에 해당하므로, 최소 1개의 전이금속 분자(예: ferrocene, [Fe(CO)₅])를 벤치마크에 포함해야 한다.

**실제 ORCA/Gaussian 출력**: `parse_output`이 실제 양자화학 출력 파일로 한 번도 테스트되지 않았다면, 이 도구는 사실상 동작 불능으로 간주해야 한다. JOSS 리뷰어는 설치 후 기능을 직접 테스트한다.

### 핵심 질문 답변: 6개 도구 중 논문 수준인 것은 몇 개인가?

**2개** (`compute_ibo`, `visualize_orbital`). `compute_partial_charges`는 조건부로 3개에 포함될 수 있다. 나머지 3개(`parse_output`, `convert_format`, `analyze_bonding`)는 추가 개발과 실제 데이터 검증이 필요하다.

## Axis 6: 시각화 파이프라인 한계

### py3Dmol 한계

**헤드리스 렌더링 불가**: py3Dmol은 JavaScript 기반(3Dmol.js)이므로 서버 사이드에서 PNG/SVG를 생성할 수 없다. 논문 figure 제작에는 브라우저에서 수동 스크린샷을 찍거나 Playwright를 사용해야 한다.

**큰 cube 파일 성능**: 현재 40³ = 64,000 격자점으로 ~843KB cube 파일을 생성하고, HTML이 ~1.7MB이다. 80³ = 512,000 격자점이면 cube ~6.7MB, HTML ~13MB로 브라우저에서 느려질 수 있다. 100³ = 1,000,000 격자점이면 ~13MB cube, ~26MB HTML이 되어 실용성이 떨어진다.

**색상/렌더링 커스터마이징**: py3Dmol에서 isovalue, 색상, 투명도 조절은 가능하나, publication-quality 렌더링(안티앨리어싱, 고해상도, 특정 시점 고정)에는 한계가 있다.

### 대안 시각화 방법

| 방법 | 장점 | 단점 | 구현 난이도 |
|---|---|---|---|
| **Playwright + py3Dmol** | 기존 파이프라인 유지, headless PNG 가능 | Playwright 설치 필요 (chromium), CI에서 무겁다 | 2/5 |
| **IboView 연동 (.molden 내보내기)** | 최고 품질, Knizia의 참조 구현 | IboView가 GUI 전용, 자동화 불가 | 3/5 (내보내기만 구현) |
| **VTK.js** | WebGL 기반, 더 나은 렌더링 | 새로운 의존성, py3Dmol 대비 설정 복잡 | 4/5 |
| **nglview** | Jupyter 통합 좋음 | 위젯 의존성 무거움, 헤드리스 더 어려움 | 3/5 |
| **matplotlib + marching cubes** | 순수 Python, PNG 직접 생성 | 인터랙티브 아님, 품질 낮음 | 2/5 |

### Cube 해상도와 논문 figure 품질

40³은 빠른 프리뷰에는 적합하지만 논문 figure에는 부족하다. 일반적으로 Gaussian의 cubegen은 기본 80점/edge, 고품질은 100~120점/edge를 사용한다. 물 분자(3원자) 수준에서 80³은 1~2초, 100³은 3~5초로 부담이 적다. 벤젠(12원자) 수준에서 100³은 10~20초, 메모리 ~100MB로 여전히 실용적이다.

**권장**: `generate_cube`에 `nx=ny=nz` 파라미터를 추가하고 기본값을 80으로 올리되, 사용자가 조절할 수 있게 해야 한다.

### 핵심 질문 답변: 현재 시각화가 논문 figure 품질에 도달하는가?

**아니다.** 40³ 해상도는 프리뷰용이며, 80³ 이상이 필요하다. 또한 py3Dmol의 브라우저 렌더링을 PNG로 변환하는 자동화 파이프라인이 없다. 최소한의 조치는 (1) 기본 해상도를 80으로 올리고, (2) Playwright 기반 HTML→PNG 스크립트를 `utils/`에 추가하는 것이다. 최상의 경로는 .molden 내보내기를 구현하여 IboView에서 최종 figure를 만드는 것인데, 이는 원래 Strategy F의 핵심이었다.

## Axis 8: 논문 전략 + 차별화 강화

### 3단 로켓 전략 현실성 재평가

**1단 JOSS**: 여전히 유효하나, 위에서 분석한 대로 collaborative effort, research impact, AI disclosure에서 상당한 준비가 필요하다. 2026-09 제출은 기술적으로 가능하나, 외부 engagement 확보가 관건. 2026-10~11 제출이 더 안전하다.

**2단 JCIM**: JOSS와 스코프 분리 가능. JCIM은 벤치마크 데이터 + LLM 워크플로우 가속 효과에 초점. 다만 JCIM은 IF 5.6의 고임팩트 저널이므로 단순 벤치마크만으로는 부족하고, LLM 에이전트가 계산화학 워크플로우를 얼마나 가속하는지의 정량적 측정(시간 단축, 오류 감소)이 필요하다. El Agente 논문(Matter, 2025)이 이미 유사한 주장을 했으므로 차별화가 어려울 수 있다.

**3단 JCTC**: 반응경로 IBO 추적은 기술적으로 가장 novelty가 높지만, 구현 복잡도도 가장 높다. 현실적으로 1단 JOSS가 성공한 후에 고려할 사안.

### 경쟁 논문과의 스코프 비교

**ChemLint vs QCViz-MCP**: 거의 겹치지 않는다. ChemLint은 SMILES/RDKit 수준의 cheminformatics(분자 fingerprint, 물성 예측), QCViz-MCP는 전자구조 수준(궤도 계산, 시각화). 오히려 ChemLint의 존재가 QCViz-MCP의 novelty를 강화한다 — "ChemLint이 분자 수준 MCP를 다루었지만 전자구조 수준은 다루지 않았다"고 말할 수 있다.

**El Agente vs QCViz-MCP**: El Agente는 범용 양자화학 에이전트(워크플로우 자동화)이고 MCP를 사용하지 않는다. QCViz-MCP는 MCP 프로토콜을 통한 도구 노출이 핵심이며, 특히 IAO/IBO에 특화. 스코프 충돌 없음.

**Argonne mcp.science**: QCViz-MCP의 가장 직접적 비교 대상. 다만 Argonne은 범용 HPC MCP 서버를 통한 PySCF 호출이고, QCViz-MCP는 전자구조 분석 전용 도구. "build vs contribute" 질문에 대한 답: Argonne 서버는 함수를 원격 실행하는 인프라 계층이고, QCViz-MCP는 IAO/IBO라는 특정 화학적 분석 로직을 내장한 도메인 계층이다.

### 차별화 강화를 위한 추가 기능 제안

1. **molden 내보내기** (export_molden): IboView와의 호환성을 실증하여 "IboView 생태계와의 연결"이라는 고유 가치를 확보.
2. **반응경로 IBO 추적** (track_ibo_along_path): IRC(Intrinsic Reaction Coordinate) 경로를 따라 IBO가 어떻게 변하는지 추적. 이것은 다른 어떤 MCP 서버도 하지 않는 기능이며, Knizia & Klein (2015)의 curly-arrow 분석을 자동화하는 것.
3. **cc-pVDZ/cc-pVTZ 벤치마크**: 실무 수준 기저함수로의 확장이 "toy example"를 넘어섰다는 증거.

### JOSS paper 핵심 레퍼런스 추가 목록

현재 6개에서 최소 12~15개로 확장 필요:

- Pan et al. "Experiences with MCP Servers for Science and HPC" arXiv:2508.18489 (2025) — **Argonne MCP 논문, 필수 인용**
- ChemLint: ChemRxiv 10.26434/chemrxiv.15000386 (2026)
- El Agente: Matter (Cell Press), 2025, DOI: 10.1016/j.matt.2025.xxx
- VASPilot: arXiv:2508.07035 (2025)
- Code2MCP: arXiv:2509.05941 (2025)
- Anthropic MCP specification (2024)
- FastMCP 3.0 (2026)
- JOSS 자체 인용: Smith (2018), JOSS design paper
- Löwdin orthogonalization 방법론 참조

### 핵심 질문 답변: novelty claim이 충분히 강한가?

**현재 상태로는 불충분하다.** "왜 이것이 기존 도구의 wrapper 이상인가?"에 대한 답변은 다음과 같아야 한다: (1) PySCF의 IAO/IBO 함수를 단순히 래핑한 것이 아니라, XYZ 자동 파싱, Löwdin 정규직교화 기반 전하 계산, cube→HTML 인터랙티브 시각화라는 분석 파이프라인을 구축했다. (2) BackendRegistry의 선택적 의존성 아키텍처가 PySCF 없는 환경에서도 파싱/변환/시각화를 제공한다. (3) MCP 프로토콜을 통해 LLM 에이전트가 자연어로 궤도 분석을 요청할 수 있는 인터페이스를 제공한다. **하지만** 이 주장을 뒷받침하려면 cc-pVDZ 이상의 벤치마크, 최소 1개의 실제 연구 워크플로우 데모, 외부 피드백이 필요하다.
