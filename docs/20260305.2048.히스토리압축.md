# QCViz-MCP 프로젝트 히스토리 (2026-03-05 기준)

## 1. 프로젝트 개요

**프로젝트명**: QCViz-MCP (Quantum Chemistry Visualization MCP Server)

**한 줄 요약**: 양자화학 전자구조 시각화(특히 Intrinsic Atomic Orbitals / Intrinsic Bond Orbitals)를 Model Context Protocol을 통해 LLM 에이전트에서 직접 호출 가능하게 하는 Python 서버.

이 프로젝트의 핵심 가치 제안은 기존 분자 구조 수준의 MCP 서버(PyMOL, ChimeraX 제어)에서 한 단계 더 깊이 들어가, 전자구조 수준의 궤도 계산과 시각화를 LLM 워크플로우에 통합한다는 것이다. 기존에는 연구자가 PySCF로 IBO를 계산하고, cube 파일을 생성하고, 별도의 시각화 도구로 열어야 했던 과정을 하나의 MCP 도구 호출로 자동화한다.

프로젝트의 대상 사용자는 계산화학 전문가이며, 특히 IboView 사용자와 PySCF 기반 워크플로우를 가진 연구자를 1차 타겟으로 한다. 최종 목표는 세 단계로 구성된다. 1단계로 JOSS(Journal of Open Source Software) 논문을 통해 소프트웨어 자체를 출판하고 DOI를 확보한다. 2단계로 J. Chem. Inf. Model.(JCIM)에 벤치마크 기반 응용 논문을 제출한다. 3단계로 선택적으로 J. Chem. Theory Comput.(JCTC)에 반응경로 IBO 자동 추적 방법론 논문을 제출한다. 이 중 1단계 JOSS가 현재 진행 중인 목표이다.

프로젝트는 2026년 3월 5일에 시작되었으며, 단독 개발자가 AI 에이전트(Claude Code, Gemini 등)의 보조를 받아 진행하고 있다. 개발 환경은 Windows(주 작업) + WSL2 Ubuntu(PySCF 실행·테스트)이며, conda(miniconda3)를 패키지 관리자로 사용한다.

## 2. 기술 스택 확정 내역

### Python 3.10+ / src-layout

프로젝트는 Python 3.10 이상을 요구하며, `src/qcviz_mcp/` 형태의 src-layout을 채택하였다. src-layout을 선택한 이유는 설치하지 않은 상태에서 소스 디렉토리가 직접 import되는 사고를 방지하고, 패키징 모범 사례를 따르기 위함이다.

### FastMCP >=3.0,<4.0

MCP 서버 프레임워크로 FastMCP를 선택하였다. 초기 설계 시점에는 FastMCP 2.x가 최신이었으나, 플랜 검증 과정에서 FastMCP 3.0이 2026년 1월 20일에 출시된 사실을 발견하였다. 3.0은 Provider/Transform 아키텍처 도입, 데코레이터가 함수를 그대로 반환(테스트 용이), state 메서드 async 전환 등의 breaking change를 포함한다. 2.x와 3.x를 동시에 지원하는 것은 위험하다고 판단하여 `fastmcp>=3.0,<4.0`으로 고정하였다. 대안으로 직접 MCP 프로토콜을 구현하는 방안이 있었으나, FastMCP의 성숙도(일일 백만 다운로드)와 데코레이터 패턴의 편의성 때문에 기각하였다.

### PySCF (IAO/IBO 계산)

핵심 궤도 계산 엔진으로 PySCF를 선택하였다. `pyscf.lo.iao`와 `pyscf.lo.ibo` 모듈이 Knizia의 IAO/IBO 알고리즘을 직접 구현하고 있으며, Gerald Knizia의 참조 구현(`ibo-ref`)과 알고리즘적으로 일치한다. Psi4도 검토하였으나 IAO/IBO 지원이 제한적이고 Python API가 PySCF보다 덜 직관적이어서 기각하였다. PySCF는 Windows를 네이티브 지원하지 않으므로 conda-forge를 통한 설치를 권장 경로로 확립하였다.

### cclib (출력 파싱)

양자화학 프로그램 출력 파싱에 cclib을 채택하였다. ORCA, Gaussian, GAMESS, NWChem, Psi4, Q-Chem 등 16개 프로그램을 지원하며, MO 계수, 에너지, 진동 모드 등을 통일된 인터페이스로 추출할 수 있다. 대안 없이 사실상 유일한 범용 파서이다.

### py3Dmol / 3Dmol.js (시각화)

시각화에는 py3Dmol(3Dmol.js의 Python 래퍼)을 선택하였다. Jupyter 노트북 친화적이고, `addVolumetricData`로 Gaussian cube 데이터를 직접 렌더링할 수 있으며, 설치가 `pip install py3Dmol` 한 줄이다. nglview도 검토하였으나 Jupyter 위젯 의존성이 더 무겁고, 헤드리스 PNG 내보내기가 py3Dmol보다 복잡하여 기각하였다.

### ASE (구조 I/O)

구조 파일 변환(xyz, cif, pdb, POSCAR 등)에 ASE를 채택하였다. 사실상 표준 도구이며 대안이 필요 없었다.

### BSD-3-Clause 라이선스

초기 문서(HISTORY02.md)에서는 MIT를 언급하였으나, 최종 작업지시 프롬프트(SECTION 2.2)에서 BSD-3-Clause로 확정하였다. PySCF(Apache-2.0), cclib(BSD-3), ASE(LGPL-2.1+), py3Dmol(MIT), FastMCP(MIT)과 모두 호환되며, JOSS의 OSI 승인 라이선스 요건을 충족한다.

## 3. Phase 1~5 설계 과정 요약

### 3.1 Phase 1: 전수조사

프로젝트의 첫 단계로 양자화학 시각화 도구 및 기존 MCP 서버에 대한 전수조사를 수행하였다. 총 18개 도구를 대상으로 Python API 유무, CLI 지원, 소켓/REST 인터페이스, 헤드리스 렌더링 가능 여부, 지원 파일 포맷, MCP 지원 여부, 라이선스를 매트릭스로 정리하였다. 조사 대상은 IboView, Multiwfn, Molden, Gabedit, Avogadro 2, PyMOL, ChimeraX, VMD, VESTA, Jmol, py3Dmol, nglview, ASE, pymatgen, cclib, 3Dmol.js, NGL Viewer, Speck이다.

IboView에 대한 심층 분석 결과, KoehnLab이 유지하는 GitHub fork(`KoehnLab/iboview`)가 존재하나 GUI 전용이며 Python API가 없고, Qt5 기반 C++ 프로그램으로 프로그래밍 방식 제어가 불가능함을 확인하였다. `.molden` 및 `.xml` 파일을 읽을 수 있으며, 내장 DFT, 실시간 IBO 시각화, 전자 흐름 다이어그램 등의 고유 기능을 보유하고 있다.

기존 MCP 서버 조사에서는 ChatMol/molecule-mcp(PyMOL·ChimeraX 제어), vrtejus/pymol-mcp(소켓 통신), XirtamEsrevni/mcp-atomictoolkit(ASE·pymatgen 기반 원자 시뮬레이션), PhelanShao의 ORCA MCP 서버(입력 파일 생성), VASPilot(VASP 다중 에이전트), ChemLint(분자 ML 워크벤치)를 확인하였다. 핵심 발견은 **전자구조 수준(궤도 시각화, 결합 분석)의 MCP 서버가 존재하지 않는다**는 것이다. 기존 서버는 모두 분자 구조 수준 또는 입력 파일 생성 수준에 머물러 있었다.

### 3.2 Phase 2: 아키텍처 설계

전수조사 결과를 바탕으로 6개 통합 전략을 평가하였다. 전략 A(GUI 자동화)는 IboView가 스크립팅을 지원하지 않아 기각하였다. 전략 B(소스코드 바인딩)는 C++ 바인딩의 높은 난이도로 조건부 보류하였다. 전략 C(파일 I/O 래핑)는 낮은 난이도이나 기능이 제한적이었다. 전략 D(PySCF를 통한 IAO/IBO)는 강력한 후보였다. 전략 E(ir-wmme 활용)는 Knizia의 적분 코드를 직접 사용하는 것으로 조건부 보류하였다. 최종 선택은 **전략 F(D+C 하이브리드)**: PySCF로 IAO/IBO를 계산하고, 결과를 `.molden` 등의 파일로 내보내 선택적으로 IboView에서 열 수 있게 하는 방식이다.

아키텍처의 핵심은 BackendRegistry 패턴이다. `registry.py`가 사용 가능한 백엔드를 자동 탐지하여 등록하며, PySCF가 없는 환경에서도 cclib와 ASE 백엔드는 정상 동작한다. ABC로 `OrbitalBackend`, `ParserBackend`, `VisualizationBackend`, `StructureBackend` 4개를 정의하였다. 초기 플랜에서는 `ComputeBackend`, `VizBackend`로 명명하였으나, 플랜 검증 과정에서 `OrbitalBackend`(서버 목적이 범용 계산이 아닌 궤도 계산), `VisualizationBackend`(약어보다 명시적)로 통일하였다.

6개 MCP 도구를 설계하였다: `compute_ibo`(IBO 계산), `visualize_orbital`(궤도 시각화), `parse_output`(양자화학 출력 파싱), `compute_partial_charges`(IAO 부분 전하), `convert_format`(구조 파일 변환), `analyze_bonding`(결합 분석).

### 3.3 Phase 3: 논문 전략

3단 로켓 전략을 수립하였다. 1단 JOSS는 소프트웨어 자체를 출판하는 짧은 논문(1-2페이지)이다. 2단 JCIM은 15-20개 벤치마크 분자에 대한 정확도·사용성·호환성 평가 논문이다. 3단 JCTC는 반응경로 IBO 자동 추적 및 curly-arrow 다이어그램 생성 방법론 논문이다.

벤치마크는 물(H₂O)부터 아스피린까지 15개 분자를 B3LYP/cc-pVDZ로 계산하여 PySCF IBO와 IboView IBO의 수치를 비교하도록 설계하였다. 메트릭은 IAO 전하 MAE <0.02 e, IBO 중첩 >0.99, 결합 차수 일관성이다.

JOSS와 JCIM의 스코프를 명확히 분리하여 자기표절을 방지하였다. JOSS는 소프트웨어 설명에 집중하고, JCIM은 과학적 평가(벤치마크 수치, LLM 워크플로우 가속 효과)에 집중한다.

### 3.4 Phase 4-5: 코드 스켈레톤 + 작업지시 프롬프트

메타-메타 프롬프트 팩토리 v1.0을 활용하여 AI 에이전트용 작업지시 프롬프트를 설계하였다. 9-SECTION 표준 구조(ROLE, TASK, CONTEXT, RESEARCH, OUTPUT, CONSTRAINTS, REASONING, SELF-CRITIQUE, EXAMPLES)를 따르며, 적응형 분기(BRANCH-FOUNDATION/SKELETON/INTEGRATION/POLISH) 로직을 포함한다. NEVER 12개, ALWAYS 12개의 제약 규칙을 정의하였다.

별도로 플랜 검증 프롬프트(7축 검증: 구조 정합성, 의존성 분석, API 설계, 테스트 전략, 누락 항목, 리스크, JOSS 요건)를 설계하여 Implementation Plan의 완전성을 사전 검증하는 체계를 갖추었다.

## 4. 구현 과정 — 실제 코딩 히스토리

### 4.1 Phase 1 구현: 프로젝트 초기화

AI 에이전트(Gemini 3.1 Pro)를 활용하여 프로젝트 기반 구조를 생성하였다. `pyproject.toml`에는 core 의존성(`fastmcp>=3.0,<4.0`, `numpy>=1.24`, `typing-extensions>=4.0`)과 5개 optional-dependencies 그룹(`compute`, `parse`, `viz`, `structure`, `all`, `dev`, `dev-no-pyscf`, `test`, `docs`)을 정의하였다.

CI 설정(`.github/workflows/ci.yml`)은 Linux에서 전체 테스트(PySCF 포함), Windows와 macOS에서 PySCF 제외 테스트를 실행하는 매트릭스로 구성하였다. ruff 설정은 `pyproject.toml` 내에 `line-length = 88`, `target-version = "py310"`, select 규칙 `["E", "F", "I", "W", "D", "UP", "B"]`로 포함하였다.

README.md, LICENSE(BSD-3-Clause), CONTRIBUTING.md를 생성하고, `tests/conftest.py`에 공통 fixture를 정의하였다. 초기 `pip install -e ".[all]"` 테스트에서 Windows 환경에서 PySCF 컴파일 실패가 발생하여 `--no-deps` 우회로 기본 테스트를 진행하였다.

### 4.2 Phase 2 구현: 백엔드

`backends/base.py`에 ABC 4개(`OrbitalBackend`, `ParserBackend`, `VisualizationBackend`, `StructureBackend`)와 데이터 클래스(`SCFResult`, `IAOResult`, `IBOResult`)를 정의하였다.

`backends/registry.py`에 `BackendRegistry`를 구현하여 `register()` 메서드로 백엔드를 등록하고, `get()` 메서드로 조회하며, 백엔드가 `is_available()` False를 반환하면 graceful하게 건너뛰도록 하였다.

`backends/pyscf_backend.py`는 `compute_scf`(RHF/UHF/RKS/B3LYP/PBE0 지원), `compute_iao`(IAO 계수 + 부분 전하), `compute_ibo`(PM 또는 Boys 국소화), `generate_cube`(미완성 — `NotImplementedError` 상태)를 구현하였다. `_HAS_PYSCF` 플래그로 PySCF 미설치 시 graceful degradation을 처리한다.

`backends/cclib_backend.py`는 cclib의 파서를 래핑하여 에너지, 좌표, 원자 심볼, MO 계수/에너지, 진동 주파수를 추출하며, `supported_programs()`로 16개 지원 프로그램 목록을 반환한다.

`backends/viz_backend.py`는 py3Dmol을 사용하여 분자 구조 HTML과 궤도 시각화 HTML을 생성한다. `backends/ase_backend.py`는 ASE의 `io.read`/`io.write`를 래핑하여 xyz, cif, pdb, POSCAR 등의 포맷 변환을 수행한다.

### 4.3 Phase 3 구현: MCP 서버

`mcp_server.py`에 FastMCP 서버를 정의하고 6개 도구를 등록하였다. `tools/core.py`에 각 도구의 실제 로직을 구현하였다. `compute_ibo`는 `atom_spec`과 `basis`를 받아 SCF → IAO → IBO 계산 후 결과 텍스트를 반환한다. `visualize_orbital`은 py3Dmol HTML을 반환한다. `parse_output`은 cclib로 파싱하여 HOMO-LUMO 갭을 포함한 요약을 반환한다.

`claude_desktop_config.json` 예시 파일도 생성하여 Claude Desktop에서 바로 연결할 수 있도록 하였다.

### 4.4 Phase 4 구현: 문서 + 논문

`paper/paper.md`를 JOSS 형식으로 작성하였다. 2025-2026년 JOSS 가이드라인 변경을 반영하여 기존 Summary, Statement of Need 외에 State of the Field, Software Design, Research Impact Statement, AI Usage Disclosure 4개 신규 필수 섹션을 추가하였다. `paper/paper.bib`에는 Knizia JCTC 2013, Knizia & Klein ACIE 2015, Sun(PySCF) 2020, O'Boyle(cclib) 2008, Larsen(ASE) 2017, FastMCP 등 6개 엔트리를 수록하였다.

## 5. 크리티컬 이슈 발견 및 해결

### 5.1 이슈 #1: FastMCP 3.0 출시 (CRITICAL)

**발견 시점**: 플랜 검증 단계에서 FastMCP 최신 버전 검색 시 발견. 2026년 1월 20일에 3.0이 출시되었으며, PyPI 최신 버전은 3.1.0이었다.

**영향 범위**: 전체 코드베이스. 데코레이터 동작 변경(함수 반환 vs Tool 객체 반환), state 메서드 async 전환, auth 설정 방식 변경 등 breaking change 다수.

**패치 내용**: `pyproject.toml`에서 `fastmcp>=3.0,<4.0`으로 버전 범위 고정. `dev-no-pyscf` extra 그룹 추가. 기존 `@mcp.tool()` 데코레이터 패턴은 3.0에서도 호환되므로 코드 변경 없이 버전만 고정하였다.

### 5.2 이슈 #2: JOSS 6개월 공개 이력 요건 (CRITICAL)

### 5.3 이슈 #3: PySCF Windows 미지원 (HIGH)

**증상**: Windows에서 `pip install -e ".[all]"` 실행 시 PySCF 컴파일 실패.

**원인**: PySCF 공식 입장 — "PySCF is not supported natively on Windows. You must use the Windows Subsystem for Linux."

**해결**: CI 매트릭스를 분리하여 `test-linux`(ubuntu-latest, Python 3.10/3.11/3.12, 전체 테스트)와 `test-non-linux`(windows/macOS, Python 3.11/3.12, `dev-no-pyscf`, `-k "not pyscf"`)로 구성하였다. 로컬 개발에서는 conda-forge를 통한 PySCF 설치를 권장 경로로 확립하였다.

### 5.4 이슈 #4: XYZ 파일 포맷 파싱 실패 (HIGH)

**증상**: WSL2 Linux에서 pytest 실행 시 PySCF 관련 4개 테스트 실패. 에러 메시지: `ValueError: Zmatrix format error at L1 Water structure`.

**원인**: 테스트 fixture `sample_water_xyz`가 표준 XYZ 파일 포맷(1행: 원자 수 "3", 2행: 코멘트 "Water structure", 3행~: 좌표)으로 되어 있었으나, PySCF의 `gto.M(atom=...)`은 이 포맷을 인식하지 못한다. PySCF는 `O 0 0 0.117; H 0 0.757 -0.469; H 0 -0.757 -0.469` 형태의 자체 포맷만 받는다.

**해결**: `_parse_atom_spec()` 헬퍼 함수를 구현하여 XYZ 포맷을 자동 감지(첫 줄이 정수인지 확인)하고 PySCF 포맷으로 변환하도록 하였다. 동시에 `compute_scf`에서 method 유효성 검증을 `gto.M()` 호출 이전으로 이동하여, `test_unsupported_method`가 분자 빌드 에러 대신 올바른 `ValueError("지원하지 않는 메서드 유형")`를 잡을 수 있도록 하였다. 이 패치로 4개 실패 중 3개가 해결되었다.

### 5.5 이슈 #5: IAO 부분 전하 합 ≠ 0 (MEDIUM)

**증상**: `test_compute_ibo_water`에서 `np.isclose(np.sum(iao_res.charges), 0.0, atol=1e-5)` 어서션 실패. 중성 분자인 물의 IAO 부분 전하 합이 0이 되어야 하는데, 0이 아닌 값이 반환되었다.

**1차 시도 (단순 Mulliken)**: 원래 구현은 AO 레이블 기반으로 IAO를 원자에 매핑하고 offset을 수동으로 관리하는 방식이었다. 결과: 전하 합 = -0.496. IAO가 비정규직교 기저라서 단순 대각 원소 합산이 올바른 population을 주지 못한다.

**2차 시도 (Cholesky 정규화)**: IAO 중첩 행렬 `S_IAO`의 Cholesky 분해로 `S^{-1/2}`를 구하고 정규직교화한 뒤 밀도 행렬을 계산하였다. 결과: 전하 합 = -1.509. 오히려 악화되었다. 원인은 IAO 중첩 행렬이 수치적으로 정밀하게 양정치(positive definite)가 아닐 때 Cholesky가 불안정하다는 점이다.

**3차 시도 (Löwdin 정규직교화, eigh 기반)**: `numpy.linalg.eigh`로 대칭 행렬의 고유값 분해를 수행하여 `S^{-1/2}`를 구하는 Löwdin 정규직교화를 적용하였다. `eigh`는 대칭 행렬에서 항상 수치적으로 안정적이다. 정규직교화된 IAO 기저에서 `proj = C_orth^T S C_occ`, `dm_iao = 2 * proj @ proj^T`로 밀도 행렬을 구성하고, 대각 원소로 population을 추출한 뒤 핵전하를 별도로 더하였다. 결과: 전하 합 ≈ 0.0 (atol=1e-5 이내). **성공**.

**핵심 교훈**: IAO는 비정규직교 기저이므로 population analysis 전에 반드시 정규직교화가 필요하다. Löwdin 방법(`eigh` 기반)이 Cholesky보다 수치적으로 안정적이다.
